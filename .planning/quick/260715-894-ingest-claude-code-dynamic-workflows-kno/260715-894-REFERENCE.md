## Source 1 — Claude Code dynamic workflows (official docs)

URL: https://code.claude.com/docs/en/workflows
Organisation: Anthropic. Retrieved: 2026-07-15.

Key content:

- A dynamic workflow is a JavaScript script that orchestrates subagents at scale. Claude
  writes the script for the task described; a runtime executes it in the background while
  the session stays responsive.
- Reach for a workflow when a task needs more agents than one conversation can coordinate,
  or when the orchestration should be codified as a script to read and rerun. Doc examples:
  codebase-wide bug sweep, 500-file migration, cross-checked research, drafting a hard plan
  from several independent angles.
- Who-holds-the-plan comparison (the docs' own table, condensed):
  - Subagents: a worker Claude spawns; Claude decides turn by turn; intermediate results in
    Claude's context; scale = a few delegated tasks per turn.
  - Skills: instructions Claude follows; Claude decides, following the prompt; results in
    Claude's context; scale same as subagents.
  - Agent teams: a lead agent supervising peer sessions; the lead decides turn by turn;
    results in a shared task list; scale = a handful of long-running peers.
  - Workflows: a script the runtime executes; THE SCRIPT decides what runs next;
    intermediate results live in script variables; the orchestration itself is repeatable;
    scale = dozens to hundreds of agents per run; resumable in the same session.
- "A workflow moves the plan into code." Claude's context holds only the final answer.
- Quality patterns: adversarial cross-review of findings before reporting; drafting a plan
  from several angles and weighing them.
- Triggers: the keyword `ultracode` in a prompt; natural-language opt-in ("use a workflow",
  "run a workflow"); `/effort ultracode` makes Claude plan a workflow for every substantive
  task in the session. Requires Claude Code v2.1.154+; ultracode effort v2.1.203+.
- Bundled workflow: `/deep-research`. Saved workflows become `/commands` from
  `.claude/workflows/` (project) or `~/.claude/workflows/` (personal); project wins on a
  name clash. Saved workflows accept input via an `args` global.
- Runtime constraints (the docs' table):
  - No mid-run user input — only agent permission prompts can pause a run. For sign-off
    between stages, run each stage as its own workflow.
  - No direct filesystem or shell access from the script itself — agents do the work; the
    script coordinates.
  - Up to 16 concurrent agents; 1,000 agents total per run.
- Subagents spawned by a workflow always run in acceptEdits mode and inherit the tool
  allowlist; file edits are auto-approved. Cost: a run can use far more tokens than
  conversation; a "Large workflow" warning fires at >25 agents or >1.5M projected tokens.
- Availability: all paid plans, Anthropic API, Bedrock, Google Cloud Agent Platform,
  Microsoft Foundry. Disable via `/config`, `"disableWorkflows": true`, or
  `CLAUDE_CODE_DISABLE_WORKFLOWS=1`.

## Source 2 — MindStudio: Claude Code 5 workflow patterns explained (blog)

URL: https://www.mindstudio.ai/blog/claude-code-5-workflow-patterns-explained
Organisation: MindStudio. Retrieved: 2026-07-15.

The article describes five structural approaches for organising Claude's agentic
capabilities, arguing pattern choice drives reliability, cost efficiency, and output
quality.

1. Sequential Flow — tasks execute in fixed order; each output feeds the next step. Use
   when the task has predictable structure with clear dependencies and failures should
   halt the process. Examples: data transformation pipelines, document processing.
2. Operator (orchestrator-subagent) — one Claude instance orchestrates and plans; separate
   tools/subagents execute delegated work. Use when coordinating multiple capabilities and
   adapting plans on intermediate results. Example: research → synthesise → report.
3. Split-and-Merge — a large task divides into independent parallel subtasks, then results
   combine. Use for high volumes of similar, independent tasks where speed matters.
   Variants: sectioning (divide by input) or voting/consensus.
4. Agent Teams — multiple specialised Claude instances with distinct roles collaborate and
   communicate. Use when tasks need different expertise types or exceed one context
   window. Example: planning agent → coding agent → review agent. The article calls this
   the most complex pattern and the hardest to debug, due to coordination overhead.
5. Headless (fully autonomous) — Claude runs without human interaction, triggered
   automatically, delivering output to downstream systems. Use when the task is
   well-understood with predictable failure modes and has been validated in interactive
   mode first.

General guidance: Claude gains agentic capability through tool use, memory management, and
multi-step reasoning; start with simpler patterns before reaching for agent teams.

## Assessment findings to carry into the notes (from the orchestrating session)

- Workflows are a run-time execution substrate, not a recommendation layer. Overlaying
  them on agens' selection machinery conflicts: no mid-run user input (breaks interactive
  intake and approval gates), scale mismatch, and a second-orchestrator anti-pattern.
- Merit sits at the output layer: the pattern index can map design patterns to the
  substrate, and a future agens change may quote that mapping.
- Taxonomy mapping for the index's Execution substrates section:
  | MindStudio pattern | Vault entry | Workflow-substrate fit |
  |---|---|---|
  | Sequential Flow | **Prompt Chaining** | Strong — script holds the step sequence |
  | Operator | **Multi-Agent Collaboration** (Orchestrator-Workers) | Strong — script is the orchestrator |
  | Split-and-Merge | **Parallelisation** | Strong — pipeline/parallel primitives |
  | Agent Teams | **Multi-Agent Collaboration** | Partial — peer sessions suit agent teams; workflows suit fan-out |
  | Headless | **Goal Setting and Monitoring** + autonomous loop | Conditional — validated-in-interactive-mode first |
