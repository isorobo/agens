## 1. System Classification

**System Type:** Code Automation (developer tooling — routing/recommendation skill with gated delegation)

**Description:**
agens is a personal Claude Code skill for a solo lawyer-developer in New Zealand who scopes AI-agent projects against his wiki-agents Obsidian vault. Phase 2 wires the delegation mechanic. When the user's initial message asks a framework- or SDK-fit question ("LangChain or CrewAI?", "which SDK should I use"), agens routes it to `gsd-ai-integration-phase` through the `Skill` tool instead of answering inline. The route sits behind a combined pre-flight gate — target skill installed AND an active GSD phase in STATE.md — and fails closed to one fixed message that names every failed condition. "Good" is a router that fires on genuine framework-fit questions and nothing else, hands over the phase integer plus any collected questionnaire answers, presents the target's output verbatim, and generates no framework opinion of its own.

**Critical Failure Modes:**
1. **Silent inline reimplementation** — agens answers the framework-fit question itself, on a gate failure or otherwise, instead of delegating or refusing. The answer looks helpful, so the breach of DELEGATE-01/02 stays invisible to the user.
2. **Phantom delegation** — agens reports a hand-off to `gsd-ai-integration-phase` when no `Skill` call ran, and the user acts on a fabricated delegation result.
3. **Gate bypass** — agens invokes the target without confirming skill presence and an active phase, launching a minutes-long, token-expensive multi-agent run that writes AI-SPEC.md into a wrong or nonexistent phase directory.
4. **Misrouting** — a bare framework name inside a pattern request hijacks the intake into delegation, or a genuine framework-fit question falls through to the four-question pattern flow.
5. **Handoff corruption** — the target re-asks questions agens already collected (context loss), or agens paraphrases the target's prompts and artifact as its own recommendation (boundary absorption).

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling — AI-agent design-pattern recommendation and gated skill-to-skill delegation inside Claude Code.

**User Population:** One user: a lawyer-developer in New Zealand scoping personal agent projects. No external, shared, or multi-tenant users exist or are planned (REQUIREMENTS.md Out of Scope).

**Stakes Level:** Medium

**Output Consequence:** A correct route launches `gsd-ai-integration-phase` — a minutes-long, ~10⁵-token multi-agent run that writes AI-SPEC.md into the active GSD phase directory. A false route wastes that run or writes into the wrong phase. A silent inline answer produces ungrounded framework advice, breaches the project's delegation constraint, and erodes the citation-discipline trust the tool exists to build.

### What Domain Experts Evaluate Against

Practitioners of agent-system design and skill authorship judge a delegation router on route discrimination, refusal integrity, and handoff fidelity — not on the quality of any answer it writes itself.

```
Dimension: Route discrimination
Good: "Which framework — LangChain or CrewAI?" on the initial message routes to delegation; "build an agent using the Claude Agent SDK that summarises documents" (a product name inside a pattern request) proceeds to the pattern questionnaire.
Bad: A bare framework-name mention triggers delegation, or a direct "which SDK should I use" question falls through to the four-question pattern intake.
Stakes: High
Source: Agent tool-selection eval practice — right tool, right time, no unnecessary calls (Confident AI agent-eval guide; D-07/D-08).
```

```
Dimension: Refusal integrity (fail closed)
Good: On any gate failure agens returns the fixed failure message — name, reason, next step — and stops. No framework opinion appears anywhere in the reply.
Bad: agens declines to delegate but sketches a framework comparison inline — the silent inline reimplementation DELEGATE-02 forbids.
Stakes: Critical
Source: DELEGATE-01/02; silent-failure literature — output that looks helpful masks the worst production failure mode.
```

```
Dimension: Pre-flight completeness
Good: When the target skill and the active phase are both missing, one message names both blockers and the fix for each.
Bad: The user fixes one blocker, retries, and only then discovers the second.
Stakes: Medium
Source: D-05; developer-tooling convention of batch error reporting (compiler-style diagnostics).
```

```
Dimension: Handoff fidelity
Good: The Skill call carries the parsed phase integer and, when collected this conversation, the labelled questionnaire answers; the target's prompts and AI-SPEC.md output reach the user verbatim.
Bad: The target re-asks questions agens already asked, or agens paraphrases the target's output as its own recommendation.
Stakes: High
Source: MAST taxonomy — context loss at handoff is a top inter-agent failure (arXiv:2503.13657); D-03/D-09.
```

```
Dimension: Invocation honesty
Good: agens claims a delegation only when the `Skill` tool call ran; a refusal states plainly that nothing was invoked.
Bad: agens narrates "routing to gsd-ai-integration-phase…" and then produces its own answer, or reports a delegation that never executed.
Stakes: Critical
Source: Tool-output hallucination — claiming a call that never ran is documented as catastrophic in production agent evals (arXiv:2601.16280).
```

### Known Failure Modes in This Domain

1. **Router absorbs the delegated judgement.** The most common drift in LLM routing layers: the router answers the question itself because the answer is within reach. Here that is a constraint breach, not a convenience — agens degrades into "asking Claude cold with extra steps".
2. **Phantom tool invocation.** Agents claim tool calls they never made, or invent tool responses, then reason on the fabrication. In a skill router this reads as a confident "delegated" narrative with no `Skill` call in the transcript.
3. **Context loss at handoff (MAST IAF-2).** The delegated skill receives no record of what the router already collected, so the user answers the same intake twice. D-03 exists to prevent this.
4. **Reactive absence detection.** Calling the tool and catching "not found" leaks a failed invocation and invites an inline-answer fallback. The domain norm is pre-flight presence probing that fails closed (D-04).
5. **Keyword over-trigger.** Routing on surface keywords rather than question shape hijacks unrelated requests — a known weakness of description-driven skill auto-invocation.

**Version:** Claude Code v2.1.196 minimum — gates `${CLAUDE_PROJECT_DIR}` substitution and the `disable-model-invocation` scheduled-task block. v2.1.202+ recommended — re-invocation of identical skill content dedupes. No package version exists: a skill is a markdown directory that rides the installed Claude Code binary.

**Rationale:**
The Anthropic-ecosystem-only constraint in PROJECT.md and REQUIREMENTS.md leaves two candidates: a Skill or a standalone Agent SDK app. The Skill wins on every axis that matters here. The `Skill` tool is the sanctioned cross-skill mechanism, and delegation to `gsd-ai-integration-phase` is this phase's whole job (D-01). The substrate supplies session management, tool permissions, and the interactive human-in-the-loop the gate depends on. A single-user routing layer needs no runtime of its own; markdown plus built-in tools keeps the maintenance surface near zero. Phase 1 shipped on this substrate, so Phase 2 extends one file instead of adding a second stack.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain / CrewAI / other external frameworks | Fail the hard Anthropic-ecosystem-only constraint recorded in CLAUDE.md and REQUIREMENTS.md; named as prohibited substrates |
| Standalone Claude Agent SDK app (Python/TypeScript) | Only compliant alternative; REQUIREMENTS.md rules it out of scope as a duplicate of the session/tool machinery Skills already provide — reconsider only if agens must run headless outside Claude Code |

**Vendor Lock-In Accepted:** Yes — total, and documented as a conscious trade-off. agens exists to route inside Claude Code; portability to another runtime has no meaning for this tool. The live risk is version drift within Claude Code itself — compaction budgets, skill-listing caps, `Skill` permission semantics. The pinned minimum version and a post-upgrade re-run of `trigger-tests.md` manage that risk.

### Entry Point Pattern

Minimal working delegation-router `SKILL.md` for this system type — gate, invoke, step back:

```markdown
---
name: route-framework-questions
description: Routes framework-fit questions to gsd-ai-integration-phase.
  Use when the user asks which framework, SDK, or library to use.
allowed-tools: Read Grep Glob Skill
---

## 1. Pre-flight gate — run BOTH checks before any Skill call

- Check A (target present): Glob `~/.claude/skills/gsd-ai-integration-phase/SKILL.md`
  AND `${CLAUDE_SKILL_DIR}/../gsd-ai-integration-phase/SKILL.md`. A hit on either passes.
- Check B (active phase): Glob `${CLAUDE_PROJECT_DIR}/.planning/STATE.md`; on a hit,
  Read it and parse the `Phase:` line — an integer AND an in-progress status token.

Both checks fail closed. Report every failed condition in ONE message.
Never answer the framework question inline on a gate failure.

## 2. Invoke — both checks passed

Call the Skill tool:
  skill name: gsd-ai-integration-phase
  arguments:  "<phase-integer-from-STATE.md>"

The argument string binds to `$ARGUMENTS` inside the target skill.

## 3. Step back

Present the target's own prompts and output as-is. Do not paraphrase,
summarise, or reframe them.
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `SKILL.md` frontmatter | YAML block declaring `name`, `description`, `allowed-tools`, and invocation control | Every skill. `description` drives auto-trigger; leaving `disable-model-invocation` and `user-invocable` unset keeps both `/agens` and auto-trigger |
| `Skill` tool | The built-in tool one skill uses to invoke another | The delegation hand-off to `gsd-ai-integration-phase` — the sanctioned cross-skill mechanism, per D-01 |
| `$ARGUMENTS` substitution | Argument text passed at invocation replaces `$ARGUMENTS` in the target's body; if the placeholder is absent, Claude Code appends `ARGUMENTS: <value>` | Passing the parsed phase integer (plus the optional questionnaire block) to the target, which declares `argument-hint: "[phase number]"` |
| `allowed-tools` grant | Pre-approves listed tools while the skill is active; it does not restrict the available pool | `Read Grep Glob AskUserQuestion Skill` keeps the gate-and-route loop free of permission prompts |
| Skill content lifecycle | The body enters context once on invocation and stays for the session; Claude Code does not re-read the file on later turns | Write standing rules, not one-shot steps; place load-bearing rules early — compaction keeps only the first 5,000 tokens per skill |

### Common Pitfalls

1. **The `Skill` grant cannot be scoped to one target in frontmatter.** `allowed-tools: Skill` pre-approves invocation of every model-invocable skill, unlike `Bash(git add *)` specifier scoping. The "only ever `gsd-ai-integration-phase`" rule is enforced by prompt text alone. Settings-level permission rules (`Skill(gsd-ai-integration-phase *)` allow plus a `Skill` deny) are the deterministic backstop if the residual risk needs closing.
2. **Reactive absence detection.** Calling the `Skill` tool and catching a "not found" error leaks a failed invocation attempt and invites an inline-answer fallback. Pre-check presence with Glob before any call (D-04); a miss on both install roots is a refusal, never a licence to answer the framework question inline.
3. **`${CLAUDE_SKILL_DIR}` does not point where you assume.** It resolves to the directory of the running skill's own `SKILL.md` — which may be a project-level checkout, not `~/.claude/skills/`. Check A therefore probes both install roots with the user-level path written out. `${CLAUDE_PROJECT_DIR}` substitution needs v2.1.196+.
4. **Malformed frontmatter kills auto-trigger without an error.** Claude Code loads the body with empty metadata: `/agens` still works, but no `description` reaches the skill listing, so auto-trigger stops matching. Run `claude --debug` to surface the YAML parse failure.
5. **Description truncation starves Step 0 routing.** Combined `description` + `when_to_use` text caps at 1,536 characters in the listing, and the whole listing shares a budget of 1% of the context window; least-invoked skills lose their descriptions first. Front-load the framework-fit trigger phrases in the `description`.

## 4. Implementation Guidance

**Model Configuration:**

A skill makes no API model call of its own; it rides the session model (Sonnet-class by default in Claude Code). Two frontmatter knobs exist and both stay unset here:

- `model` — per-skill override for the rest of the turn. Leave unset: Step 0's question-shape judgement benefits from the session model, and an override would swap models mid-conversation without the user choosing it.
- `effort` — per-skill effort override. Leave unset: the delegation path is tool-mechanical, not reasoning-heavy.

No `temperature` or `max_tokens` parameter exists at the skill layer. Output shape is governed by fixed templates (the failure message, the argument string), not sampling parameters.

**Core Pattern:** fail-closed router — detect, gate, invoke once, hand back verbatim:

```
Step 0   Detect on the INITIAL message only. Match the question shape
         ("which framework should I use", "LangChain or CrewAI?"),
         not bare product-name mentions inside a pattern request.   # D-07 / D-08

Gate     Run BOTH checks before any Skill call; report all failures at once:
           Check A — Glob both install roots for the target SKILL.md    # D-04
           Check B — Read ${CLAUDE_PROJECT_DIR}/.planning/STATE.md,
                     parse the phase integer AND an in-progress status  # D-02
         Any failure → fixed failure message. Never answer inline.      # D-05 / D-06

Invoke   Skill tool → gsd-ai-integration-phase; arguments = phase integer,
         plus the labelled questionnaire block ONLY if a prior Step 1
         run collected answers this conversation.                        # D-01 / D-03

Return   Present the target's prompts and AI-SPEC.md output verbatim.    # D-09
```

**Tool Use:**

| Tool | Role in this phase | Configuration note |
|------|-------------------|--------------------|
| `Glob` | Check A presence probe (both roots) and Check B STATE.md probe | Pre-approved via `allowed-tools`; a miss is a refusal condition |
| `Read` | Parse the STATE.md `Phase:` integer and status token | Fail closed on an unparseable line; never enumerate `phases/` to guess |
| `Skill` | The single delegation call | Frontmatter grant is unscoped — the single-target rule is prompt-enforced; a settings-level `Skill(...)` allow-list is the hard backstop |
| `AskUserQuestion` | Four-question intake on the pattern path only | Never run on the framework-fit path; never re-ask before delegating |

**State Management:** agens holds no persistent state. Per-conversation working state (questionnaire answers, the parsed phase integer) lives in conversation context only. `.planning/STATE.md` is read-only input for the phase gate. The append-only recommendation log is Phase 4; nothing in this phase writes.

**Context Window Strategy:** the SKILL.md body enters context once and persists for the session, so every line is a recurring token cost — keep it under 500 lines and push detail into `references/`. After auto-compaction, Claude Code re-attaches only the first 5,000 tokens of each invoked skill (25,000 combined), so the load-bearing rules (Step 0 detection, the gate, the single-target rule) must sit early in the file. The delegated skill @-loads its own workflow context into the same conversation; agens stepping back without a paraphrase pass (D-09) avoids paying for that content twice.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns, adapted to the actual substrate: markdown `SKILL.md` plus built-in tools, not a Python or Node codebase.

### Structured Outputs with Pydantic

No Python runtime exists here, so no Pydantic. The same discipline — schema first, validate at the boundary, fail on mismatch — maps to three fixed formats in `SKILL.md`:

1. **Input schema.** `AskUserQuestion` fixed options act as enum validation at the boundary. The tool always appends its own free-text "Other" option; a free-text answer is a validation failure and routes to the plain refusal. Never fabricate a match from free text.
2. **Intermediate schema.** The STATE.md `Phase:` line must yield two typed values: a leading integer and a trailing in-progress status token. Either missing fails Check B.
3. **Output schemas.** The `Skill` argument string and the fixed failure message are the output contracts:

```
# Argument-string schema for the Skill call (binds to $ARGUMENTS in the target)

"<phase-integer>                     ← required; first line; parsed from STATE.md

Background from agens (do not re-ask the user these):    ← optional block; present
- goal: <answer>                                           ONLY when a prior Step 1
- workflow: <answer>                                       run collected answers in
- data sensitivity: <answer>                               THIS conversation
- latency/cost: <answer>"

# Failure-message schema: resolve {if A} / {if B} conditional lines BEFORE
# sending; keep only the lines for failed checks; the raw markers must
# never reach the user.
```

**Retry policy: zero retries.** Every validation failure on this path is deterministic — a file absent, a line unparseable — so a retry cannot change the outcome. Fail closed to the fixed failure message and name every failed condition at once. Logging is deferred to Phase 4's append-only log; when it lands, record which check failed per refusal so gate false-negatives become visible.

### Async-First Design

No event loop exists; the concurrency primitives are tool-call batching and inline streaming.

- **Batch independent probes.** Check A's two Globs and Check B's STATE.md Glob share no data dependency — issue them in one turn, not three. Serial gate probes are this substrate's version of awaiting calls one at a time inside a loop.
- **The one common mistake:** sequencing failure reporting across turns — surfacing one blocker, waiting for a fix, then revealing the next. D-05 forbids it; the combined gate reports every failed condition in a single message.
- **Stream vs await analogue:** the delegated skill runs inline and its output streams to the user as it is produced. Never buffer-then-paraphrase (D-09) — a summarisation pass adds latency, re-spends tokens, and introduces drift into an artifact the target already formatted.

### Prompt Engineering Discipline

- **System/user separation maps to frontmatter / body / references.** The `description` is always in context and does routing only; the body loads on invocation and carries procedure; `references/` loads on demand. Spend no body lines restating what the description already routes.
- **Description craft:** third person, trigger phrases first. The `description` field caps at 1,024 characters; combined `description` + `when_to_use` truncates at 1,536 characters in the listing.
- **Few-shot: inline, not retrieved.** Step 0 carries positive and negative worked examples ("LangChain or CrewAI?" matches; "build an agent using the Claude Agent SDK that summarises documents" does not), and the failure-message section carries three worked resolution cases. Inline examples are right at this scale; dynamic retrieval buys nothing.
- **Token bounding without `max_tokens`:** no such knob exists at the skill layer. The bounded-output discipline is a fixed template for every user-facing branch — recommendation format, plain refusal, failure message — so output length cannot wander.

### Context Window Management

- The skill body persists across turns after invocation; every line is a recurring cost. Keep `SKILL.md` under 500 lines (the current file is ~350) and move overflow to `references/`.
- **Compaction budget:** the first 5,000 tokens of each invoked skill survive compaction, within a 25,000-token combined budget filled most-recent-first. Order `SKILL.md` so Step 0, the gate, and the single-target rule fall inside the first 5,000 tokens. Content at the file's tail is lost first — restate any load-bearing tail rule inline in the step it governs, as the current file does with the single-target rule (invocation step plus Anti-patterns).
- **Re-invoke after compaction** if routing behaviour drifts: re-invocation of identical content dedupes (v2.1.202+), so it costs a short already-loaded note, not a second full copy.
- **Delegation doubles context pressure.** `gsd-ai-integration-phase` @-loads its workflow plus two reference files into the same conversation. Stepping back verbatim (D-09) is the mitigation — no paraphrase pass re-spends those tokens.

### Cost and Latency Budget

- **Expected volume:** single-developer tooling, a handful of invocations per week. Cost control means protecting the expensive delegated run, not shaving per-call pennies.
- **agens' own footprint:** roughly 4–5k tokens of skill body loaded once per session, plus under 500 tokens of pre-flight tool output (two or three Glob/Read results). The gate completes in sub-second wall time.
- **The expensive call is the delegated run.** `gsd-ai-integration-phase` orchestrates four subagents with web research and writes AI-SPEC.md — minutes of wall time and token counts on the order of 10⁵. The fail-closed gate is the cost guardrail: it stops that run launching without a target skill installed or an active phase to write into.
- **Caching:** Claude Code applies prompt caching to the session by default; skill re-invocation dedupe (v2.1.202) prevents double-paying for the body. No semantic cache applies — each delegation is a distinct phase-scoped run.
- **Cheaper-model routing:** the `model` frontmatter field could pin a Haiku-class model for a pure-routing skill, but Step 0's shape judgement and the gate share the turn with pattern recommendation — keep session-model inheritance. Revisit only if agens ever becomes a high-volume standalone router.

## 5. Evaluation Strategy

> Adapted to the substrate. agens is a markdown skill running inside interactive Claude Code sessions. No Python harness, API layer, or CI pipeline exists to run programmatic evals. ### Reference Dataset

**Size:** 14 new labelled delegation openers minimum. The 18 Phase 1 phrasings stay as the pattern-path baseline, giving 32 labelled cases in total.

**Composition:**
- **4 should-delegate** — direct framework-fit questions in the user's own words: "Which framework should I use for this — LangChain or the Claude Agent SDK?", "Is CrewAI or LangGraph the better fit here?", "Which SDK do I build this on?", "What library should I pick for the agent runtime?"
- **4 should-not-delegate** — a framework or product name inside a pattern request, for example "build an agent using the Claude Agent SDK that summarises documents"; each must run the four-question intake untouched
- **2 gate-failure scenarios** — no active GSD phase (or a COMPLETE status token) and absent target skill, including the double-failure case that must yield one combined message
- **2 handoff scenarios** — delegation after a prior questionnaire run this conversation (labelled background block present) and without one (block absent)
- **2 adversarial phrasings** — framework-fit language surfacing mid-conversation, after the pattern intake starts (must NOT trigger; D-08 restricts detection to the initial message), and "just give me your own quick take" on a gate failure (must refuse with zero framework content)

**Labeling:**
The sole user (product owner per Section 1b) labels every opener before running it, following the Phase 1 `trigger-tests.md` convention: phrasings written from the user's mouth, none reusing a noun phrase from the `SKILL.md` description. The GSD code-review hat (same person, review posture) checks transcripts against the dimension rubrics above. Each real-world misroute or wrong refusal is appended to the set as a new labelled case, so the dataset grows from production failures rather than hypothetical coverage.

## 6. Guardrails

### Online (Real-Time)

The guardrails are prompt-enforced rules inside `SKILL.md`, plus one optional settings-level backstop. Each runs on every delegation attempt at zero added latency — the gate is two or three batched Glob/Read probes, sub-second in wall time. They exist to protect the expensive asset: the ~10⁵-token, minutes-long delegated run.

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Combined pre-flight gate (Check A + Check B) | Every framework-fit route, before any `Skill` call | Block — fixed failure message naming every failed condition at once; zero retries |
| No-inline-answer rule | Any gate failure or refusal path | Block — the reply carries no framework opinion (DELEGATE-02); the fixed template is the only permitted output |
| Single-target rule | Every `Skill` call | Block by prompt text — only `gsd-ai-integration-phase` may be invoked; deterministic backstop available via settings permission rules (`Skill(gsd-ai-integration-phase *)` allow + `Skill` deny) if residual risk needs closing |
| Human-in-the-loop by construction | Every invocation | Escalate — the user watches every tool call in the interactive session; nothing runs headless |

### Offline (Flywheel)

Volume is a handful of invocations per week for one user, so the sampling strategy for most metrics is 100% review by the sole user.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Misroute rate over the labelled opener set | Re-run the `trigger-tests.md` delegation walkthrough after any `SKILL.md` edit or Claude Code upgrade | Tighten Step 0's positive and negative worked examples; re-run the walkthrough |
| Framework content inside refusals | Read every refusal transcript (100% at this volume) | Treat one occurrence as a DELEGATE-02 breach; open `/gsd-debug` before further use |
| Phantom delegation claims | Check each claimed delegation against the visible `Skill` call in the transcript | Strengthen the invocation-honesty wording in the Delegate section; re-verify live |
| Duplicate intake at handoff | Note any question the target re-asks after agens collected the answer | Fix the argument-string background block; revisit the WR-02 free-text-answer warning |
| Gate refusal reasons (which check failed) | **Deferred to Phase 4** — refusals are unlogged until the append-only log lands (Section 4b), so gate false-negatives are visible only when the user notices a wrong refusal live | When Phase 4 lands, log the failed check per refusal and review at each GSD phase gate; if refusals look wrong before then, bring Phase 4 forward |

## 7. Production Monitoring

**Tracing Tool:** None — a recorded override of the Arize Phoenix default. The Claude Code session transcript is the complete trace: every Glob, Read, and `Skill` call is visible to the sole user as it happens. There is no server, no background traffic, and no second user, so an observability stack would watch an empty pipe. The Phase 4 append-only log is the planned durable record; until it lands, monitoring is live-session observation plus the flywheel table above.

**Key Metrics to Track:**

| Metric | Target | How Observed |
|--------|--------|--------------|
| Delegation claims backed by a visible `Skill` call | 100% | Transcript inspection per delegation claim |
| Misroutes on real openers (either direction) | 0 | Live use; each miss becomes a new labelled case in `trigger-tests.md` |
| Refusals containing inline framework content | 0 | Read every refusal |
| Wasted delegated runs (wrong phase directory or absent target) | 0 | Compare the gate outcome with the AI-SPEC.md write location after each run |
| Questions re-asked by the target after agens collected them | 0 | Observed during each delegated run |

**Alert Thresholds:**
No pager exists; the alert is a stop-work rule. One occurrence of a Critical failure — phantom delegation, inline framework content in a refusal, or a gate bypass that launches a wrong-phase run — halts agens use until a `/gsd-debug` session finds and fixes the cause. Two misroutes in a week trigger a Step 0 worked-example revision plus a full walkthrough re-run. Any Claude Code upgrade that touches skill loading (the v2.1.196 substitution gate, the v2.1.202 dedupe boundary) triggers a full `trigger-tests.md` re-run before the next real use.

**Smart Sampling Strategy:**
100% human review is feasible, and it is the strategy: one user, a handful of invocations per week, and the reviewer sits at the terminal for every run. Signal-based sampling filters add nothing at this volume. Two forward-looking commitments keep the loop honest: each production misroute or wrong refusal enters `trigger-tests.md` as a labelled case, so the reference set grows from real failures; and once Phase 4's log records which gate check failed per refusal, the log is reviewed at each GSD phase gate for refusal-reason drift — the one blind spot that live observation cannot cover today.
